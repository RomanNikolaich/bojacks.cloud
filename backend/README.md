# Backend  - Облачное хранилище файлов

## 📖 Описание
Сервер для веб-приложения для загрузки и управления файлами в облаке. 
Поддерживает регистрацию, авторизацию, загрузку файлов, их изменение, скачивание и удаление.

## 🛠 Технологии
- Язык: Python 3.11+
- Фреймворк: Django 6.x + Django REST Framework
- База данных: PostgreSQL 17
- ORM: Django ORM
- Аутентификация: JWT (SimpleJWT) + Refresh Tokens
- WSGI-сервер: Gunicorn
- Reverse-proxy: Nginx

## 🚀 Развёртывание в продакшене (Nginx + Gunicorn)
В продакшене используется связка Nginx + Gunicorn, где Nginx выступает единой точкой входа:
- Nginx (порт 80/443) — отдаёт статические ресурсы и проксирует API-запросы
- Gunicorn (WSGI-сервер) — обрабатывает Django-приложение

### Архитектура

Поскольку проект использует архитектуру SPA (React + Django REST API), бэкенд не выполняет серверные редиректы. Вместо этого после успешной регистрации API возвращает ID созданного пользователя в JSON-ответе, а фронтенд (React) самостоятельно выполняет клиентский редирект на страницу профиля пользователя. Это соответствует принципам REST и разделению ответственности между frontend и backend.


| URL   | Обработчик                    |
|-------|-------------------------------|
| /     | Статика фронтенда (Vite build) |
| /api/* | Прокси на Gunicorn (Django REST Framework) |
| /admin/ | Прокси на Gunicorn (Django admin) |
| /static/ | Статика Django (collectstatic) |
| /media/ | Загруженные пользователями файлы |

## 🔗 CORS (для разработки)

В режиме разработки фронтенд работает на `http://localhost:5173`, 
а бэкенд на `http://localhost:8000`. Для корректной работы 
междоменных запросов используется библиотека `django-cors-headers`:

```python
# settings.py
CORS_ALLOWED_ORIGINS = [
    "http://localhost:5173",  # Vite dev server
]
```

В продакшене CORS не требуется, так как фронтенд и бэкенд
обслуживаются одним доменом через Nginx.

### Описание приложений

Проект разделен на два Django-приложения, каждое из которых отвечает за свою область ответственности (принцип Separation of Concerns):

#### 📦 Приложение `users`

**Назначение:** Управление пользователями, аутентификация и авторизация.

**Ответственность:**
- Регистрация новых пользователей (`UsersSignUpView`)
- Аутентификация и выдача JWT-токенов (`UsersLogInView`)
- Административные операции с пользователями (создание, изменение ролей, удаление)
- Управление правами доступа (`is_staff`, `is_superuser`)

**Модели:**
- `User` — расширенная модель пользователя (наследник `AbstractUser`)

**Ключевые компоненты:**
| Файл | Назначение |
|------|-----------|
| `models.py` | Модель пользователя с дополнительными полями |
| `views.py` | View-классы для регистрации, входа, админских операций |
| `serializers.py` | Сериализаторы для валидации данных пользователей |
| `signals.py` | Сигналы для автоматического создания профиля при регистрации |

**Почему отдельное приложение:**
Логика управления пользователями не зависит от логики работы с файлами. Это позволяет:
- Легко тестировать аутентификацию отдельно
- Переиспользовать приложение в других проектах
- Изменять модель пользователя без влияния на файловое хранилище


#### 📦 Приложение `users_files`

**Назначение:** Управление файлами пользователей — загрузка, хранение, скачивание, удаление.

**Ответственность:**
- Загрузка файлов на сервер (`AddFilesView`)
- Получение списка файлов пользователя (`GiveFilesView`)
- Изменение метаданных файлов (`PatchFilesView`)
- Удаление файлов (`DeleteFilesView`)
- Публичный доступ к файлам по специальным ссылкам (`DownloadFileByTokenView`, `OpenFileByTokenView`)

**Модели:**
- `UsersFiles` — модель файла с метаданными (имя, размер, дата загрузки, специальный токен)

**Ключевые компоненты:**
| Файл | Назначение |
|------|-----------|
| `models.py` | Модель файла с полями для хранения метаданных |
| `views.py` | View-классы для CRUD-операций с файлами |
| `serializers.py` | Сериализаторы для валидации файлов и их данных |
| `permissions.py` | Кастомные разрешения (например, `IsOwnerOrAdmin`) |
| `signals.py` | Сигналы для автоматического удаления файлов с диска при удалении из БД |

**Почему отдельное приложение:**
Файловое хранилище — это самостоятельная подсистема со своей бизнес-логикой. Разделение позволяет:
- Независимо развивать функционал работы с файлами
- Легко добавить новые типы файлов или хранилища (например, S3)
- Не смешивать логику пользователей и файлов


### 🔗 Взаимодействие приложений

Приложения связаны через **внешние ключи** (Foreign Keys):

```python
# В users_files/models.py
class UsersFiles(models.Model):
    user = models.ForeignKey(
        User, 
        on_delete=models.CASCADE,
        related_name='files'  # Позволяет обращаться к файлам через user.files.all()
    )
```

Это означает:
- Каждый файл принадлежит одному пользователю
- При удалении пользователя все его файлы удаляются автоматически (CASCADE)
- Пользователь может получить список своих файлов через user.files.all()

## Аутентификация и отслеживание состояния
Состояние аутентификации пользователя отслеживается через JWT-токены (JSON Web Tokens). При успешной регистрации или входе пользователю выдаются два токена:
Access token — для доступа к API (короткий срок жизни)
Refresh token — для обновления access token (длительный срок жизни)
При каждом запросе к защищенным эндпоинтам клиент должен передавать access token в заголовке:

- Authorization: Bearer <access_token>

Сервер проверяет подпись токена и срок действия. Если токен действителен — запрос выполняется, иначе возвращается ошибка 401 Unauthorized.


## Проверка прав доступа

Все API-вызовы проверяют права доступа пользователя через механизмы Django REST Framework:
Аутентификация: Защищенные эндпоинты требуют наличия валидного JWT-токена в заголовке Authorization: Bearer <token>. При отсутствии или невалидности токена возвращается ошибка 401 Unauthorized в формате JSON.
Авторизация: Административные эндпоинты требуют прав администратора (is_staff или is_superuser). При попытке доступа обычного пользователя возвращается ошибка 403 Forbidden в формате JSON.
Object-level permissions: При попытке изменить или удалить чужой файл возвращается ошибка 403 Forbidden (или 404 Not Found для сокрытия факта существования ресурса).
Публичные эндпоинты: Эндпоинты регистрации, входа и скачивания файлов по специальной ссылке не требуют аутентификации, но проверяют валидность токена/ссылки.

Примеры ошибок:

// 401 Unauthorized
{"detail": "Authentication credentials were not provided."}

// 403 Forbidden
{"detail": "You do not have permission to perform this action."}

// 404 Not Found

## Логирование событий сервера

Все события сервера логируются с использованием стандартного модуля Python logging. Логи выводятся в консоль и содержат:
- Уровень события (DEBUG, INFO, WARNING, ERROR, CRITICAL)
- Дату и время события
- Модуль, в котором произошло событие
- Детальную информацию для анализа

Примеры логируемых событий:
- Загрузка, скачивание, удаление файлов (INFO)
- Регистрация и вход пользователей (INFO)
- Ошибки валидации (WARNING)
- Отсутствие файлов на диске (ERROR)
- Попытки несанкционированного доступа (WARNING)

Настройка уровня логирования:
В settings.py можно изменить уровень детализации:
- DEBUG — максимальная детализация (для разработки)
- INFO — стандартный режим
- WARNING — только предупреждения и ошибки (для production)
